# /deli:test — тесты по бизнес-требованию

Создай или обнови реальные тесты проекта по файлу бизнес-требований `docs/business-requirements/FEATURE_KEY.md`.

Если проект или бизнес-требование относится к S2T/data mapping, используй S2T-сценарий: возьми Excel-файл из папки `s2t`, прочитай лист `Target_columns` и на его основе создай SQL-тесты в CSV-формате. Если Excel нельзя прочитать напрямую, временно конвертируй нужный лист в CSV, используй этот CSV как источник и удали временный CSV после создания тестов.

## Использование

```text
/deli:test BR-001
```

## Когда использовать

Используй `/deli:test`:

- после ручного заполнения `docs/business-requirements/BR-XXX.md`;
- перед `/deli:run`, если тесты нужно подготовить отдельно;
- внутри реализации фичи до review;
- когда пользователь просит добавить тесты по требованию.

## Алгоритм

1. Найди файл требований:

```text
docs/business-requirements/FEATURE_KEY.md
```

Если файл не найден, остановись и предложи сначала выполнить `/deli:feature FEATURE_KEY "Описание задачи"`.

2. Прочитай в требовании секции:
   - `## Что и зачем`
   - `## Требования`
   - `## Критерии готовности`
   - `## Требования к тестам`
   - `## Ограничения`

3. Если в файле есть `TODO`, unchecked-пункты `- [ ]` или `status: draft`, не останавливайся автоматически. Сначала оцени, можно ли создать полезные тесты по уже имеющейся информации.

   Если информации недостаточно, задай вопросы пользователю по одному:

```text
Вопрос 1/N: [конкретный вопрос о тестируемом поведении]
```

   Спрашивай только то, что нужно для тестов:
   - основной сценарий;
   - edge cases;
   - ожидаемые ошибки;
   - входные и выходные данные;
   - права доступа;
   - интеграции и внешние зависимости;
   - какие тесты обязательны: unit, integration, API, UI, E2E.

4. Определи тестовый стек проекта по существующим файлам:
   - `pom.xml`, `build.gradle`, `build.gradle.kts`, `build.sbt` — JVM/Scala/Java/Kotlin;
   - `package.json` — Node.js/TypeScript/JavaScript;
   - `pyproject.toml`, `requirements.txt`, `pytest.ini` — Python;
   - `go.mod` — Go;
   - `Cargo.toml` — Rust;
   - `*.csproj`, `*.sln` — .NET;
   - существующие папки `test`, `tests`, `src/test`, `__tests__`, `features`.

5. Следуй существующим паттернам проекта:
   - расположение тестов;
   - naming convention;
   - test framework;
   - fixtures/mocks/builders;
   - стиль assert-ов;
   - как в проекте обычно оформляют test doubles и фикстуры.

6. Если для фичи нужен S2T/data mapping SQL-набор тестов, выполни отдельный S2T-сценарий:
   - найди Excel-файл в папке `s2t`;
   - прочитай лист `Target_columns`;
   - если Excel нельзя прочитать напрямую, создай временный CSV только из листа `Target_columns`, прочитай данные из него и после работы удали этот временный CSV;
   - не выводи в чат содержимое Excel/CSV, названия чувствительных полей, значения данных, SQL с чувствительными литералами или другие чувствительные данные;
   - сформируй CSV-файл тестов со столбцами ровно `Id:Entity:Test_name:Test_script:Expected_result`;
   - разделитель выходного CSV должен соответствовать принятому в проекте формату; если формат не определен, используй двоеточие как разделитель, потому что заголовок задан как `Id:Entity:Test_name:Test_script:Expected_result`;
   - SQL в `Test_script` должен быть самодостаточным запросом проверки и не должен содержать реальных чувствительных значений из данных.

   Обязательные типы SQL-тестов:
   - кардинальность по всем целевым таблицам: отдельный тест на каждую таблицу, проверяющий ожидаемое количество строк или допустимое правило подсчета, если оно задано в S2T;
   - заполненность обязательных атрибутов: для каждого атрибута с `col null = NO` создай тест, который проверяет отсутствие `NULL`;
   - дубли по первичным ключам: для каждого PK создай тест, который группирует по PK и находит ключи с `count(*) > 1`;
   - битые ссылки по внешним ключам: для каждого FK определи соответствующий PK и создай тест, который проверяет, что каждому FK находится соответствующий PK.

   Если соответствие FK -> PK нельзя надежно определить из `Target_columns`, остановись и задай пользователю конкретный вопрос по этой связи. Не угадывай связи, от которых зависит корректность SQL-теста.

   Рекомендуемые `Expected_result`:
   - для проверок нарушений используй ожидание `0 rows`;
   - для кардинальности используй конкретное ожидаемое значение или правило из S2T, если оно есть; если его нет, явно укажи, что ожидание требует заполнения человеком.

7. Создай или обнови тесты, которые покрывают:
   - must-have требования;
   - критерии готовности;
   - негативные сценарии и edge cases;
   - регрессии, явно вытекающие из требования.

8. Не создавай отдельную git-ветку. Тесты относятся к той же feature branch и тому же PR.

9. Если тесты невозможно создать без реализации production-кода, создай минимальные pending/skipped тесты только если это принято в проекте. Иначе зафиксируй список тест-кейсов в ответе и объясни, какой production-код должен появиться перед добавлением тестов.

10. Не запускай тесты. На этой стадии достаточно создать или обновить тестовые файлы. Если видишь очевидную команду проверки, можешь указать ее в отчете как справочную, но не выполняй.

11. После создания тестов удали все временные файлы, созданные для анализа или конвертации, включая временный CSV из Excel. Не удаляй исходный Excel-файл из `s2t` и не удаляй итоговый CSV-файл с тестами.

12. После работы выведи отчет без чувствительных данных:

```text
═══════════════════════════════════════════
ТЕСТЫ ОБНОВЛЕНЫ
═══════════════════════════════════════════

Feature: FEATURE_KEY
Requirements: docs/business-requirements/FEATURE_KEY.md

Измененные тесты:
  - path/to/test_1
  - path/to/test_2

Покрытие:
  - [критерий/требование]
  - [edge case]

Команда проверки:
  - не запускалась; тесты только созданы или обновлены

Временные файлы:
  - удалены

Следующий шаг:
  → /deli:validate feature FEATURE_KEY
  → /deli:run FEATURE_KEY
═══════════════════════════════════════════
```

13. Зафиксируй прохождение стадии в state:

```powershell
python "%USERPROFILE%\.gigacode\extensions\delion\main.py" --project-root "%CD%" /deli:mark FEATURE_KEY tests_created --test-result "tests updated"
```

## Важно

- Не заменяй существующий тестовый стиль проекта своим.
- Не удаляй существующие тесты без явной причины.
- Не создавай тесты вне принятой структуры проекта.
- Не запускай локальные тесты в рамках `/deli:test`.
- Не переходи к review/CI, если обязательные тесты не созданы и нет явного обоснования.
- Не выводи чувствительные данные из S2T/Excel/CSV в чат или отчет.
- После S2T-сценария удаляй временные файлы, включая CSV, созданный только для чтения Excel.
