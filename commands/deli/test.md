# /deli:test — тесты по бизнес-требованию

Создай или обнови реальные тесты проекта по файлу бизнес-требований `docs/business-requirements/FEATURE_KEY.md`.

## Использование

```text
/deli:test BR-001
```

## Когда использовать

Используй `/deli:test`:

- после ручного заполнения `docs/business-requirements/BR-XXX.md`;
- перед `/deli:run`, если тесты нужно подготовить отдельно;
- внутри реализации фичи до review;
- когда пользователь просит добавить тесты по требованию.

## Алгоритм

1. Найди файл требований:

```text
docs/business-requirements/FEATURE_KEY.md
```

Если файл не найден, остановись и предложи сначала выполнить `/deli:feature FEATURE_KEY "Описание задачи"`.

2. Прочитай в требовании секции:
   - `## Что и зачем`
   - `## Требования`
   - `## Критерии готовности`
   - `## Требования к тестам`
   - `## Ограничения`

3. Если в файле есть `TODO`, unchecked-пункты `- [ ]` или `status: draft`, не останавливайся автоматически. Сначала оцени, можно ли создать полезные тесты по уже имеющейся информации.

   Если информации недостаточно, задай вопросы пользователю по одному:

```text
Вопрос 1/N: [конкретный вопрос о тестируемом поведении]
```

   Спрашивай только то, что нужно для тестов:
   - основной сценарий;
   - edge cases;
   - ожидаемые ошибки;
   - входные и выходные данные;
   - права доступа;
   - интеграции и внешние зависимости;
   - какие тесты обязательны: unit, integration, API, UI, E2E.

4. Определи тестовый стек проекта по существующим файлам:
   - `pom.xml`, `build.gradle`, `build.gradle.kts`, `build.sbt` — JVM/Scala/Java/Kotlin;
   - `package.json` — Node.js/TypeScript/JavaScript;
   - `pyproject.toml`, `requirements.txt`, `pytest.ini` — Python;
   - `go.mod` — Go;
   - `Cargo.toml` — Rust;
   - `*.csproj`, `*.sln` — .NET;
   - существующие папки `test`, `tests`, `src/test`, `__tests__`, `features`.

5. Следуй существующим паттернам проекта:
   - расположение тестов;
   - naming convention;
   - test framework;
   - fixtures/mocks/builders;
   - стиль assert-ов;
   - как в проекте обычно оформляют test doubles и фикстуры.

6. Создай или обнови тесты, которые покрывают:
   - must-have требования;
   - критерии готовности;
   - негативные сценарии и edge cases;
   - регрессии, явно вытекающие из требования.

7. Не создавай отдельную git-ветку. Тесты относятся к той же feature branch и тому же PR.

8. Если тесты невозможно создать без реализации production-кода, создай минимальные pending/skipped тесты только если это принято в проекте. Иначе зафиксируй список тест-кейсов в ответе и объясни, какой production-код должен появиться перед добавлением тестов.

9. Не запускай тесты. На этой стадии достаточно создать или обновить тестовые файлы. Если видишь очевидную команду проверки, можешь указать ее в отчете как справочную, но не выполняй.

10. После работы выведи отчет:

```text
═══════════════════════════════════════════
ТЕСТЫ ОБНОВЛЕНЫ
═══════════════════════════════════════════

Feature: FEATURE_KEY
Requirements: docs/business-requirements/FEATURE_KEY.md

Измененные тесты:
  - path/to/test_1
  - path/to/test_2

Покрытие:
  - [критерий/требование]
  - [edge case]

Команда проверки:
  - не запускалась; тесты только созданы или обновлены

Следующий шаг:
  → /deli:validate feature FEATURE_KEY
  → /deli:run FEATURE_KEY
═══════════════════════════════════════════
```

11. Зафиксируй прохождение стадии в state:

```powershell
python "%USERPROFILE%\.gigacode\extensions\delion\main.py" --project-root "%CD%" /deli:mark FEATURE_KEY tests_created --test-result "tests updated"
```

## Важно

- Не заменяй существующий тестовый стиль проекта своим.
- Не удаляй существующие тесты без явной причины.
- Не создавай тесты вне принятой структуры проекта.
- Не запускай локальные тесты в рамках `/deli:test`.
- Не переходи к review/CI, если обязательные тесты не созданы и нет явного обоснования.
