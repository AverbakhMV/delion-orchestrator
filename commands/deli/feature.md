# /deli:feature — создать бизнес-требование

Создай легковесный файл бизнес-требований для фичи, которую можно вести через Delion workflow без отдельного PRD.

Перед анализом и созданием бизнес-требования прочитай корневой `AGENTS.md` целевого проекта, если файл существует. Учитывай описанные в нем правила, ограничения, стиль работы и проектный контекст при формулировке требований, критериев готовности, ограничений и требований к тестам.

## Использование

```text
/deli:feature BR-001 "Добавить экспорт отчета в CSV"
/deli:feature BR-001 @docs/input/feature-request.md
```

## Когда использовать

Используй `/deli:feature`, когда пользователь просит:

- завести новую фичу;
- описать небольшое или среднее изменение;
- подготовить требования перед реализацией;
- создать `BR-XXX` для дальнейшего `/deli:run`.

Если изменение выглядит крупным, рискованным или затрагивает архитектуру, создай бизнес-требование, но явно предложи дополнить техническую спецификацию в `docs/specs/` перед запуском реализации.

## Уточняющие вопросы

Перед созданием файла требований выполни короткий анализ исходного описания:

1. Сформулируй, что уже понятно:
   - что нужно изменить;
   - зачем это нужно;
   - кто пользователь или потребитель результата;
   - какой основной сценарий;
   - какие ограничения, риски и зависимости видны;
   - какие критерии готовности и тесты можно вывести из описания.

2. Составь список неясностей. Вопросы нужны, если не хватает данных для:
   - бизнес-ценности;
   - пользователей или ролей;
   - основного happy path;
   - edge cases и ошибок;
   - границ фичи: что входит и что не входит;
   - acceptance criteria;
   - требований к тестам;
   - интеграций, прав доступа, данных или миграций;
   - приоритета `P1/P2/P3`;
   - размера `S/M/L`.

3. Если есть неясности, не создавай файл сразу. Задавай вопросы пользователю по одному.

   Формат:

```text
Вопрос 1/N: [конкретный вопрос]
```

4. После каждого ответа обновляй список оставшихся неясностей и задавай следующий вопрос.

5. Если пользователь отвечает, что деталей пока нет, зафиксируй это в будущем документе как `TODO`, но явно скажи, что такой файл не пройдет `/deli:validate` до заполнения.

6. Когда вопросы закончились или пользователь явно просит создать черновик с `TODO`, переходи к созданию файла.

## Алгоритм

1. Определи ключ требования:
   - если пользователь передал `BR-XXX`, используй его;
   - если ключ не передан, предложи следующий свободный ключ вручную, например `BR-001`.

2. Проверь наличие `AGENTS.md` в корне целевого проекта:
   - если файл есть, прочитай его до анализа исходного описания;
   - если файла нет, продолжай работу, но явно отметь пользователю, что проектный `AGENTS.md` не найден.

3. Проанализируй исходное описание с учетом `AGENTS.md` и проведи цикл уточняющих вопросов по правилам выше.

4. Перед созданием файла проверь, что не существует:

```text
docs/business-requirements/BR-XXX.md
```

Если файл уже есть, не перезаписывай его без явного подтверждения пользователя.

5. Если пользователь передал путь к файлу, выполни:

```powershell
python "%USERPROFILE%\.gigacode\extensions\delion\main.py" --project-root "%CD%" /deli:feature FEATURE_KEY @path/to/file.md
```

6. Если пользователь передал текст задачи, выполни:

```powershell
python "%USERPROFILE%\.gigacode\extensions\delion\main.py" --project-root "%CD%" /deli:feature FEATURE_KEY "Текст задачи"
```

   В `Текст задачи` включи исходное описание и ответы пользователя на уточняющие вопросы, чтобы runtime создал более полный черновик.

7. После создания файла проверь, что он находится в:

```text
docs/business-requirements/FEATURE_KEY.md
```

8. Попроси человека проверить и дополнить документ перед запуском workflow.

## Шаблон файла бизнес-требований

```markdown
---
id: BR-XXX
title: "[Краткое название]"
status: draft
created: YYYY-MM-DD
priority: P2
size: M
---

# Delion: бизнес-требования BR-XXX

## Исходная задача

[Текст задачи или ссылка на файл-источник]

## Что и зачем

**Что делаем:**
[Кратко описать изменение]

**Зачем:**
[Бизнес-ценность. Если неочевидно — TODO]

**Для кого:**
[Пользователи, роли или системы-потребители. Если неочевидно — TODO]

## Требования

### Должно быть (Must have)

- [ ] [Основное требование из описания]

### Хорошо бы (Nice to have)

- [ ] [Дополнительные пожелания, если пользователь их упомянул]

## Критерии готовности

- [ ] Функционал работает для основного сценария.
- [ ] Ошибки и edge cases обработаны.
- [ ] Поведение покрыто созданными или обновленными тестами либо явно описано, почему тесты не нужны.

## Требования к тестам

- [ ] Указать acceptance criteria, которые должны быть покрыты автоматическими тестами.
- [ ] Указать обязательные негативные сценарии и edge cases.
- [ ] Указать, какие тестовые файлы и сценарии должны быть созданы или обновлены для этой фичи.

## Ограничения

- Одна фича выполняется в одной feature-ветке и одном PR.
- Внутренние work items не создают отдельные ветки.
- Code review должен быть принят до push и PR.
- PR не создается до прохождения review loop.
- Jenkins CI запускается после merge PR в основную ветку.
- TODO: добавить проектные ограничения, если есть.

## Готовность к разработке

- [ ] Требования проверены человеком.
- [ ] Must have требования полные и проверяемые.
- [ ] Критерии готовности полные и проверяемые.
- [ ] Требования к тестам заполнены.
- [ ] Системные требования проекта учтены.
- [ ] Инструкции из корневого `AGENTS.md` учтены.

## Следующий шаг

После ручной проверки:

```text
/deli:validate feature BR-XXX
/deli:run BR-XXX
```
```

## Важно

- Не запускай `/deli:run`, пока файл содержит `TODO`, unchecked-пункты или статус `draft`.
- Если исходное описание неполное, сначала задай уточняющие вопросы по одному и только потом создавай файл.
- Для size `M` или `L` предложи заполнить техническую спецификацию в `docs/specs/`.
- Для size `L` явно предупреди, что фича должна быть разбита на внутренние work items до реализации; work items фиксируются в `.deli/state.json` через `/deli:subtask`, но выполняются в одной feature branch и одном PR.
