# /deli:review — code review по бизнес-требованию

Проведи code review изменений по `FEATURE_KEY` и добейся, чтобы замечания review были исправлены основным агентом в той же feature branch.

## Использование

```text
/deli:review BR-001
```

## Когда использовать

Используй `/deli:review`:

- после реализации фичи и обновления тестов;
- перед push, созданием PR и merge;
- после исправления замечаний, чтобы повторить review;
- когда пользователь просит проверить изменения по бизнес-требованию.

## Обязательные входные данные

- `docs/business-requirements/FEATURE_KEY.md`
- изменения в рабочем дереве или текущей feature branch
- тесты, созданные или обновленные по `/deli:test FEATURE_KEY`

Если файл требований не найден, остановись и предложи сначала выполнить `/deli:feature FEATURE_KEY "Описание задачи"`.

## Алгоритм

1. Прочитай бизнес-требование:

```text
docs/business-requirements/FEATURE_KEY.md
```

Особенно проверь:

- `## Требования`
- `## Критерии готовности`
- `## Требования к тестам`
- `## Ограничения`
- `## Готовность к разработке`

2. Проверь текущее состояние git:

```text
git status
git diff
git diff --staged
```

Если изменений нет, остановись и сообщи, что review невозможен без diff.

3. Проведи review как отдельная роль code reviewer.

Ищи в первую очередь:

- blocker: код не компилируется, ломает запуск, теряет данные, нарушает безопасность;
- high: требование не реализовано, неверная бизнес-логика, отсутствуют обязательные тесты;
- medium: edge case не покрыт, ошибка обработки исключений, риск регрессии;
- low: читаемость, дублирование, локальные улучшения без риска.

4. Проверь соответствие изменениям:

- must-have требования покрыты кодом;
- критерии готовности проверяемы;
- тесты реально покрывают требования и edge cases;
- нет отдельной ветки для внутренних work items;
- изменения не выходят за scope `FEATURE_KEY`;
- нет случайных unrelated правок;
- нет секретов, токенов, локальных путей и временных файлов.

5. Сформируй review findings.

Формат каждого замечания:

```text
[severity] path:line — краткий заголовок
Проблема: что не так.
Риск: чем это может сломаться.
Исправление: что должен изменить основной агент.
```

6. Если есть замечания `blocker`, `high` или `medium`, основной агент обязан внести правки в код в той же feature branch.

Правила исправления:

- не создавай новую branch;
- не откатывай чужие или unrelated изменения;
- исправляй минимально достаточным изменением;
- обнови или добавь тесты, если замечание связано с поведением;
- не запускай локальные тесты после правок; достаточно обновить тестовые файлы;
- затем повтори `/deli:review FEATURE_KEY`.

7. Review loop считается завершенным только если:

- нет `blocker/high/medium` замечаний;
- оставшиеся `low` замечания либо исправлены, либо явно приняты как не блокирующие;
- тесты для измененного поведения есть или есть явное обоснование, почему они не нужны.

8. После завершения выведи итог.

Если замечания найдены:

```text
═══════════════════════════════════════════
CODE REVIEW: CHANGES REQUESTED
═══════════════════════════════════════════

Feature: FEATURE_KEY
Requirements: docs/business-requirements/FEATURE_KEY.md

Findings:
  [high] path/to/file:42 — ...
  [medium] path/to/test:18 — ...

Required fixes:
  - ...
  - ...

Next:
  → основной агент исправляет замечания в той же ветке
  → /deli:test FEATURE_KEY при изменении поведения
  → /deli:review FEATURE_KEY повторно
═══════════════════════════════════════════
```

Если review принят:

```text
═══════════════════════════════════════════
CODE REVIEW: ACCEPTED
═══════════════════════════════════════════

Feature: FEATURE_KEY
Requirements: docs/business-requirements/FEATURE_KEY.md

Checked:
  ✓ соответствие требованиям
  ✓ тесты
  ✓ edge cases
  ✓ отсутствие unrelated changes

Next:
  → push branch
  → create PR
  → merge PR
  → /deli:ci FEATURE_KEY
═══════════════════════════════════════════
```

После принятия review зафиксируй стадию в state:

```powershell
python "%USERPROFILE%\.gigacode\extensions\delion\main.py" --project-root "%CD%" /deli:mark FEATURE_KEY reviewed --review-summary "review accepted"
```

## Важно

- Review-agent не должен молча исправлять код вместо основного агента.
- Основной агент вносит правки по замечаниям review и запускает повторный review.
- Не переходи к push, PR и merge, пока review loop не принят.
- Jenkins CI запускается только после merge PR в основную ветку.
